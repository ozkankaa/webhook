Architecture Overview — What is Hot Chocolate?

Foundation

Hot Chocolate is an open-source GraphQL server framework for ASP.NET Core. Unlike Hasura, it is a code-first library — not a platform you install beside your services. It lives inside your ASP.NET Core application process, sharing your DI container, middleware pipeline, authentication stack, and deployment unit. You define the GraphQL schema in C# and Hot Chocolate serves it as a spec-compliant GraphQL endpoint.

Two schema-building approaches

ApproachDescriptionBest for
Implementation-firstC# classes + attributes ([QueryType], [MutationType]). Source generator creates schema at build time. Recommended for new code.Most scenarios — thin, idiomatic C#
Code-firstFluent type descriptor API to define schema shape independently of C# modelsWhen GraphQL shape must diverge from domain model; infrastructure/generator tooling

Fusion — the distributed layer (optional)

Hot Chocolate Fusion is a separate composition layer that turns multiple independent Hot Chocolate servers into a federated supergraph behind a single gateway. It implements the open GraphQL-Fusion specification (MIT licence). A single HC server starts without Fusion — you add it when you need team-level ownership boundaries or independent deployment of subgraphs.

Key positioning for Intelliflo: Every existing ASP.NET Core REST service can add a GraphQL endpoint by installing one NuGet package and calling app.MapGraphQL(). You do not need to rewrite anything, split infrastructure, or learn a new DSL. REST endpoints and GraphQL coexist in the same process, sharing auth middleware, EF Core contexts, and DI configuration.

Developer Productivity & Experience

Native .NET Fit
◆ Strong fit — stays entirely within your existing C# / ASP.NET Core skillset

What Hot Chocolate gives your team out of the box

  • IQueryable middleware pipeline — annotate a resolver with [UsePaging], [UseFiltering], [UseSorting], [UseProjection] and HC auto-generates the full GraphQL filtering, sorting, pagination input types and translates them into native EF Core / SQL Server queries. No hand-written SQL or filter parsers.
  • Cursor-based pagination — Relay Connections spec out of the box; SQL Server and SQLite null-ordering detected automatically. Offset pagination also supported via [UseOffsetPaging].
  • DataLoader (Green Donut) — built-in N+1 solution. Declare a DataLoader class; HC batches all resolver calls for that entity in a single request into one database round-trip.
  • Nitro IDE — local, embedded GraphQL IDE. No cloud account needed. Schema explorer, query history, subscription testing, performance breakdown per field.
  • Schema SDL export — commit schema.graphql to Git on every build to get diff-reviewed schema changes in PRs.
  • Trusted Documents — register approved operations at build time; server rejects anything not pre-approved. Critical for L1 API control.
  • Cost Analysis — assign field weights, define a query budget. Operations exceeding the budget are rejected before execution. Essential for public/partner APIs.

Developer workflow — adding a new entity

1. EF Core entity — your existing pattern
public class Portfolio
{
    public Guid PortfolioId { get; set; }
    public Guid ClientId   { get; set; }
    public Guid TenantId   { get; set; }
    public string  Name       { get; set; } = "";
    public decimal Valuation  { get; set; }
    public Client  Client     { get; set; } = null!;
}
2. Query resolver — four attributes give you filter, sort, page, project
[QueryType]
public class PortfolioQueries
{
    [UsePaging(MaxPageSize = 100, IncludeTotalCount = true)]
    [UseProjection]
    [UseFiltering]
    [UseSorting]
    public IQueryable<Portfolio> GetPortfolios(
        [Service] AppDbContext db,
        [GlobalState("currentTenantId")] Guid tenantId)
        => db.Portfolios
              .Where(p => p.TenantId == tenantId)
              .AsNoTracking();

    public async Task<Portfolio?> GetPortfolioById(
        Guid id,
        PortfolioByIdDataLoader dataLoader,
        CancellationToken ct)
        => await dataLoader.LoadAsync(id, ct);
}

Learning curve

  • Engineers already know C#, ASP.NET Core, and EF Core — the three things HC builds on.
  • GraphQL concepts (queries, mutations, subscriptions, DataLoader) are the main new knowledge area — typically 1 week to proficiency.
  • HC's attribute-based API ([UsePaging], [UseFiltering], [Authorize]) follows the same convention-over-configuration pattern your team uses in ASP.NET Core MVC.

AI-Assisted Development & AI Enablement

Strong for Code Gen

AI-assisted development (developer tooling)

C# is one of the best-represented languages in LLM training data. AI coding tools (GitHub Copilot, Cursor, Claude Code) generate correct Hot Chocolate code reliably because the HC attribute API is idiomatic, well-documented, and follows established .NET conventions. Concretely, you can prompt an AI assistant to:

  • Generate a complete QueryType resolver with paging, filtering, and sorting from an EF Core entity class.
  • Write DataLoader implementations from the entity type and repository interface.
  • Produce MutationType input records, error union types, and mutation resolvers.
  • Generate ASP.NET Core authorization policy classes from plain-English access requirements.
  • Write xUnit tests for resolvers using HC's in-process testing infrastructure.

AI enablement for your product

Hot Chocolate exposes a spec-compliant GraphQL endpoint with full introspection. Any AI agent framework (LangChain, Semantic Kernel, Claude function calling, GPT-4o tool use) can introspect your schema and call it as a structured tool. This is the standard GraphQL path — no vendor-specific MCP integration required, though you can build one.

Gap vs Hasura DDN: Hot Chocolate does not have a built-in PromptQL or MCP server equivalent. If AI agents querying your financial data in natural language is a near-term product feature, you will need to build this plumbing yourself (a Semantic Kernel skill or custom MCP server that wraps your HC endpoint). This is achievable but is additional work compared to Hasura's first-class AI integration.

Schema as AI context

Your HC schema SDL (the committed schema.graphql file) can be embedded into AI agent prompts as context for tool calling. Because HC's types are named from your C# domain classes, the schema reads as a semantic description of your domain — Clients, Portfolios, Policies — giving AI agents meaningful type names to reason over.

Maintainability, Governance & Migration Path

Strong Fit

Schema as code — Git-native governance

Your GraphQL schema is derived from your C# code — it lives in the same repository, is compiled by the same CI pipeline, and changes are reviewed in the same PRs as your business logic. Exporting the schema SDL to a file on every build (schema.graphql) gives you a human-readable, diff-able representation for code review.

Program.cs — export schema.graphql on build
// Export schema SDL for Git tracking and change review
var executor = await app.Services.GetRequestExecutorAsync();
await File.WriteAllTextAsync("schema.graphql", executor.Schema.ToString());
await app.RunAsync();

Breaking change detection

Integrate GraphQL Inspector or Rover CLI in your CI pipeline to diff schema.graphql between commits. Configure it to fail the build on breaking schema changes — removed fields, changed types, renamed arguments. This gives you the same safety net as Hasura's schema registry, assembled from open-source tooling.

GitHub Actions — schema diff step
- name: Check schema breaking changes
  run: |
    npx @graphql-inspector/cli diff \
      git:origin/main:schema.graphql \
      schema.graphql \
      --rule suppressRemovalOfDeprecatedField

Multi-Tenancy

Developer-Enforced
⚡ Fully supported — but requires disciplined implementation, not declarative config

Recommended patterns for financial-grade multi-tenancy

Pattern 1 — EF Core Global Query Filters (preferred)

Register a global filter on every tenant-scoped entity in OnModelCreating. Every LINQ query automatically includes the tenant predicate — WHERE TenantId = @tenantId — without any resolver code.

AppDbContext.cs — global tenant filter
protected override void OnModelCreating(ModelBuilder builder)
{
    // Every query on Client/Portfolio/Policy automatically scoped
    builder.Entity<Client>()
        .HasQueryFilter(c => c.TenantId == _tenantProvider.CurrentTenantId);

    builder.Entity<Portfolio>()
        .HasQueryFilter(p => p.TenantId == _tenantProvider.CurrentTenantId);

    builder.Entity<Policy>()
        .HasQueryFilter(p => p.TenantId == _tenantProvider.CurrentTenantId);
}
TenantProvider.cs — extract tenantId from JWT claim
public class TenantProvider(IHttpContextAccessor http) : ITenantProvider
{
    public Guid CurrentTenantId
    {
        get
        {
            var claim = http.HttpContext?.User
                .FindFirst("tenant_id")?.Value
                ?? throw new UnauthorizedAccessException("No tenant claim");
            return Guid.Parse(claim);
        }
    }
}
Program.cs — register scoped so each request gets its own instance
builder.Services.AddHttpContextAccessor();
builder.Services.AddScoped<ITenantProvider, TenantProvider>();
builder.Services.AddDbContext<AppDbContext>(opts =>
    opts.UseSqlServer(connectionString));

Pattern 2 — Resolver-level guard (defence in depth)

[QueryType]
public class ClientQueries
{
    [UsePaging][UseProjection][UseFiltering][UseSorting]
    public IQueryable<Client> GetClients(
        [Service] AppDbContext db,
        [Service] ITenantProvider tenant)
        // Global query filter already applies — explicit filter is defence in depth
        => db.Clients.Where(c => c.TenantId == tenant.CurrentTenantId);
}
Important limitation: EF Core's global query filters are defined at OnModelCreating time, which runs once per application lifetime, not per request. The _tenantProvider must be a scoped service resolved at query time (not captured in the closure), and the DbContext must be registered as scoped (the default). Validate this carefully — it is a known footgun. Consider integration tests that assert cross-tenant data cannot be returned.

Performance & Operations

Full Control

N+1 — DataLoader (Green Donut)

DataLoader is HC's solution to the N+1 problem. It batches all resolver invocations for the same entity type within a single GraphQL execution into one database call. HC v15/16 introduced a new DataLoader API with improved ergonomics and automatic scope management.

ClientByIdDataLoader.cs
public class ClientByIdDataLoader(
    IDbContextFactory<AppDbContext> dbFactory,
    IBatchScheduler scheduler,
    DataLoaderOptions options)
    : BatchDataLoader<Guid, Client>(scheduler, options)
{
    protected override async Task<IReadOnlyDictionary<Guid, Client>>
        LoadBatchAsync(IReadOnlyList<Guid> keys, CancellationToken ct)
    {
        await using var db = await dbFactory.CreateDbContextAsync(ct);
        return await db.Clients
            .Where(c => keys.Contains(c.ClientId))
            .ToDictionaryAsync(c => c.ClientId, ct);
    }
}

IQueryable push-down — filtering at the database

When resolvers return IQueryable<T>, HC's middleware stack ([UseFiltering], [UseSorting], [UsePaging], [UseProjection]) builds and appends LINQ expressions to the queryable before it is executed. EF Core translates the final LINQ expression into a single optimised SQL query with WHERE, ORDER BY, OFFSET/FETCH, and SELECT of only the requested columns. The database does the heavy lifting — not the application layer.

Caching

HC has no built-in response cache, but unlike Hasura you have full control over caching in C#:

  • DataLoader cache — per-request entity cache built into every DataLoader. Prevents redundant DB hits within a single operation.
  • IMemoryCache / IDistributedCache (Redis) — inject into resolvers. Cache expensive aggregations, reference data lookups, or valuation results with explicit TTL and invalidation logic.
  • EF Core second-level cache — via EFCoreSecondLevelCacheInterceptor (open source). Transparent query result caching with Redis backing.
  • Response caching at API Gateway — CloudFront, AWS API Gateway, or Azure API Management can cache GraphQL GET requests for public/low-sensitivity queries.

Subscriptions — real-time

HC supports GraphQL subscriptions over WebSockets (graphql-ws) and Server-Sent Events (SSE). The ITopicEventSender interface is injected into mutations to publish events. Backing providers:

  • In-memory — default, single-process.
  • Redis — AddRedisSubscriptions() for multi-instance pub/sub.
  • Azure Service Bus / Kafka — custom IMessageSerializer and topic bridge.

HC v16 implements @defer and @stream draft directives for incremental delivery of large datasets.

Authorisation — Identity, Claims & Scopes

Native ASP.NET Core
✓ Zero IdP changes — plugs directly into your existing auth middleware and OAuth2 scopes

Authentication setup — same as any ASP.NET Core API

Program.cs — JWT Bearer with your existing IdP (Azure AD / Entra ID)
builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(opts =>
    {
        opts.Authority = "https://login.microsoftonline.com/{tenant-id}/v2.0";
        opts.Audience  = "api://intelliflo-l1-api";
        // Standard ASP.NET Core JWT validation — no Hasura namespace injection needed
    });

builder.Services.AddAuthorization(opts =>
{
    // OAuth2 scope policies — your existing scopes work directly
    opts.AddPolicy("portfolio:read",  p => p.RequireClaim("scope", "portfolio:read"));
    opts.AddPolicy("portfolio:write", p => p.RequireClaim("scope", "portfolio:write"));
    opts.AddPolicy("admin",           p => p.RequireRole("IntellifloAdmin"));
});

Field-level visibility — hide sensitive fields per role

Client type — NI number hidden from adviser role
[ObjectType<Client>]
public static class ClientType
{
    static void Configure(IObjectTypeDescriptor<Client> descriptor)
    {
        descriptor.Field(c => c.ClientId).Type<UuidType>();
        descriptor.Field(c => c.FullName);

        // NI number visible to admin role only
        descriptor.Field(c => c.NiNumber)
            .Authorize("admin");
    }
}

Bulk & Batch Operations

Full .NET Control
✓ Best-in-class — full EF Core and SQL Server bulk operation support in C#

Bulk inserts

MutationType — bulk portfolio insert with EF Core
[MutationType]
public class PortfolioMutations
{
    [Authorize(Policy = "portfolio:write")]
    public async Task<BulkResult> BulkCreatePortfolios(
        List<CreatePortfolioInput> inputs,
        [Service] AppDbContext db,
        [Service] ITenantProvider tenant,
        CancellationToken ct)
    {
        var portfolios = inputs.Select(i => new Portfolio {
            PortfolioId = Guid.NewGuid(),
            TenantId    = tenant.CurrentTenantId,
            ClientId    = i.ClientId,
            Name        = i.Name
        }).ToList();

        db.Portfolios.AddRange(portfolios);   // single INSERT with batching
        await db.SaveChangesAsync(ct);

        return new BulkResult(portfolios.Count);
    }
}

High-performance bulk operations — EFCore.BulkExtensions

Bulk update / upsert via EFCore.BulkExtensions (SQL Server optimised)
// BulkUpdate — generates MERGE or batched UPDATE, not row-by-row
await db.BulkUpdateAsync(valuations, new BulkConfig {
    PropertiesToInclude = [nameof(Portfolio.Valuation), nameof(Portfolio.ValuedAt)],
    BatchSize = 500
});

// EF Core 7+ ExecuteUpdateAsync — set-based UPDATE without loading entities
await db.Portfolios
    .Where(p => portfolioIds.Contains(p.PortfolioId))
    .ExecuteUpdateAsync(s => s
        .SetProperty(p => p.ValuedAt, DateTime.UtcNow)
        .SetProperty(p => p.Valuation, p => p.Valuation * 1.05m), ct);

Stored procedure calls

Calling SQL Server stored procedure from mutation resolver
[Authorize(Policy = "portfolio:write")]
public async Task<BulkValuationResult> BulkUpdateValuations(
    List<ValuationInput> inputs,
    [Service] AppDbContext db,
    CancellationToken ct)
{
    var json = JsonSerializer.Serialize(inputs);
    var param = new SqlParameter("@ValuationData", json);

    var result = await db.Database
        .ExecuteSqlRawAsync("EXEC sp_BulkUpdateValuations @ValuationData", param, ct);

    return new BulkValuationResult(result);
}

HTTP request batching

HC supports HTTP request batching (array of GraphQL operations in a single POST) out of the box. Enable on the server:

app.MapGraphQL().WithOptions(new GraphQLServerOptions {
    EnableBatching = true
});

Transaction control

Full transaction management in C# — exactly the patterns your team uses today:

  • db.Database.BeginTransactionAsync() for explicit transactions spanning multiple mutations.
  • Unit of Work pattern via your existing service layer — mutations delegate to services that manage transaction boundaries.
  • Saga / outbox pattern via MediatR + your preferred outbox library for distributed transaction consistency.

public class AppDbContext(DbContextOptions<AppDbContext> opts,
                            ITenantProvider tenant)
    : DbContext(opts)
{
    public DbSet<Tenant>    Tenants    => Set<Tenant>();
    public DbSet<Client>    Clients    => Set<Client>();
    public DbSet<Portfolio> Portfolios => Set<Portfolio>();

    protected override void OnModelCreating(ModelBuilder b)
    {
        // Global query filters — every query scoped to current tenant
        b.Entity<Client>()
         .HasQueryFilter(c => c.TenantId == tenant.CurrentTenantId);

        b.Entity<Portfolio>()
         .HasQueryFilter(p => p.TenantId == tenant.CurrentTenantId);

        // Stored procedure registration (for EF mapping)
        b.Entity<BulkValuationResult>().HasNoKey().ToView(null);
    }
}

Add the DataLoader for N+1 resolution

DataLoaders/ClientByIdDataLoader.cs
public class ClientByIdDataLoader(
    IDbContextFactory<AppDbContext> dbFactory,
    IBatchScheduler scheduler,
    DataLoaderOptions? options = null)
    : BatchDataLoader<Guid, Client>(scheduler, options)
{
    protected override async Task<IReadOnlyDictionary<Guid, Client>>
        LoadBatchAsync(IReadOnlyList<Guid> keys, CancellationToken ct)
    {
        await using var db = await dbFactory.CreateDbContextAsync(ct);
        return await db.Clients
            .Where(c => keys.Contains(c.ClientId))
            .ToDictionaryAsync(c => c.ClientId, ct);
    }
}

[MutationType]
public class PortfolioMutations
{
    // Bulk insert mutation
    [Authorize(Policy = "portfolio:write")]
    public async Task<List<Portfolio>> BulkAddPortfolios(
        List<CreatePortfolioInput> inputs,
        [Service] AppDbContext db,
        [Service] ITenantProvider tenant,
        CancellationToken ct)
    {
        var portfolios = inputs.Select(i => new Portfolio {
            PortfolioId = Guid.NewGuid(),
            TenantId    = tenant.CurrentTenantId,
            ClientId    = i.ClientId,
            Name        = i.Name
        }).ToList();
        db.Portfolios.AddRange(portfolios);
        await db.SaveChangesAsync(ct);
        return portfolios;
    }

    // Stored procedure call
    [Authorize(Policy = "portfolio:write")]
    public async Task<int> BulkUpdateValuations(
        List<ValuationInput> inputs,
        [Service] AppDbContext db,
        CancellationToken ct)
    {
        var json  = JsonSerializer.Serialize(inputs);
        var param = new SqlParameter("@ValuationData", json);
        return await db.Database
            .ExecuteSqlRawAsync("EXEC sp_BulkUpdateValuations @ValuationData", param, ct);
    }
}

    public Portfolio OnValuationUpdated([EventMessage] Portfolio portfolio)
        => portfolio;

Overall verdict for Intelliflo: Hot Chocolate is the natural choice for an ASP.NET Core engineering team migrating incrementally from REST L1 APIs. It slots into your existing codebase, your existing auth pipeline, your existing EF Core patterns, and your existing deployment infrastructure with minimal disruption. The trade-offs — CRUD boilerplate per entity and developer-enforced tenancy — are well-understood .NET problems with established mitigation patterns, not architectural blockers.
