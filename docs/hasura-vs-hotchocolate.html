Architecture Decision · Intelliflo L1 Data Access Layer

Hasura DDN v3 vs Hot Chocolate v16
Head-to-Head Comparison

Evaluating both technologies against your ten criteria — with your ASP.NET Core stack, SQL Server, and financial-services governance requirements as the lens. Includes pros, cons, and a hybrid recommendation.
Hasura DDN v3 vs Hot Chocolate v16 (ChilliCream)
Hasura DDN

Zero-code data access platform

Declarative, polyglot, infrastructure-led. Eliminates CRUD boilerplate for any data source via configuration. Best when you want speed-to-API and minimal application code for the data layer.

Hot Chocolate (ChilliCream)

Code-first .NET GraphQL server framework

C#-native, EF Core-integrated, fully composable with your existing ASP.NET Core pipeline. Best when your team values full code ownership, type safety, and deep .NET ecosystem integration.

AI-Assisted Development & AI Enablement

AspectHasura DDNHot Chocolate

Slight edge — C# is the most well-represented language in LLM training data. HC's attribute-based API is idiomatic and AI tools generate correct HC code reliably.
AI agent data plane
(product feature)		 Winner — PromptQL + MCP server are first-class, roadmap-backed features. Supergraph is the natural semantic layer for AI agents. GraphQL introspection works for AI agents, but no dedicated MCP integration or natural-language query capability in the roadmap.

If AI-assisted software development means "help engineers write code faster", both are comparable. If AI enablement means "let AI agents query and reason over your financial data securely at runtime", Hasura has a structural advantage.

Multi-Tenancy

PatternHasura DDNHot Chocolate
Row-level isolation Winner — declarative session variable filter in HML. Zero code, enforced at the engine level before SQL generation. Implement in resolver/EF Core query: .Where(x => x.TenantId == currentTenantId). Must be applied consistently by developers in every resolver — human error risk.
Governance / accidental bypass Structurally safer — engine enforces filter; developers cannot accidentally expose cross-tenant data in a query without explicitly removing the permission rule. Developer discipline required. A missing .Where() clause is a security gap. Mitigated with middleware, EF Core global query filters, or custom base types — but requires intentional architecture.
Schema-per-tenant Separate connector/subgraph per tenant. Operational overhead scales. Dynamic schema compositing or DbContext per-tenant factory patterns via DI. More complex but fully within .NET idioms.
Tenant context from JWT Tie — x-hasura-tenant-id claim used directly in permission rules. Tie — extract claim from IHttpContextAccessor in C# and use throughout resolvers. Standard ASP.NET Core pattern.
For a financial services platform where a single missing tenancy filter is a data breach, Hasura's engine-enforced row-level permissions have a structural safety advantage over developer-applied EF Core filters.

Authorisation — Identity, Claims & Scopes

AspectHasura DDNHot Chocolate
Auth mode JWT (JWKS / fixed secret) or webhook. Decodes x-hasura-* claims. Multiple auth modes simultaneously (v4). Winner for .NET teams — standard ASP.NET Core AddAuthentication(JwtBearer). Integrates with your existing middleware chain, policies, and IAuthorizationService.
OAuth2 scopes Scopes must be mapped to Hasura roles in IdP token transform. Not natively scope-aware. Winner — ASP.NET Core policy-based auth supports scope claims natively. [Authorize(Policy = "portfolio:read")] on resolvers or field middleware.
Field-level auth TypePermissions in HML — specify allowed fields per role. Declarative and reviewed in PRs. HC field-level auth via [Authorize] attribute on resolver methods or custom field middleware. C# policies enforced per field.
Row-level auth Winner — engine-enforced WHERE clause injection. Structurally impossible to bypass without changing HML. EF Core global query filters or explicit .Where() in resolvers. Effective but developer-enforced.
IdP compatibility Any JWKS-compliant IdP. Azure AD, Entra ID, Auth0, Cognito — all work with a one-time token transform to add Hasura namespace. Winner — standard Microsoft.AspNetCore.Authentication.JwtBearer. Zero configuration changes to your IdP. Uses standard JWT claims directly.
Critical difference for Intelliflo: Hot Chocolate plugs directly into your existing ASP.NET Core auth stack — AddAuthentication, AddAuthorization, and policy definitions that already govern your REST APIs. No IdP reconfiguration, no scope-to-role mapping, no new auth config format to learn. Hasura requires a one-time JWT namespace injection in your IdP — a minor but real onboarding cost.

Pros & Cons — Given Your ASP.NET Core Stack

Hasura DDN — Pros

  • Zero CRUD code — introspect SQL Server, add a model, done. No C# resolver, no EF entity, no repository. Huge productivity gain for net-new data-only entities.
  • Engine-enforced multi-tenancy — row-level isolation is a compile-time declarative guarantee, not developer discipline. Structurally safer for financial data.
  • Immutable builds + schema registry — every API version is traceable, rollback is instant, schema governance is a platform feature not a DIY CI script.
  • AI data plane (PromptQL / MCP) — if Intelliflo is building AI features, the supergraph as a governed semantic layer for AI agents is a real differentiator.
  • Polyglot future-proofing — as your data estate grows (Snowflake, MongoDB, etc.), Hasura unifies them in one API without adding .NET services per source.
  • Query pushdown — single SQL per GraphQL operation regardless of traversal depth. No DataLoader boilerplate to maintain.

Hasura DDN — Cons (in your context)

  • Not C# / not .NET — business logic lives in TypeScript/Python Commands. Your .NET engineers add a polyglot runtime they must learn, maintain, test, and deploy separately.
  • External process — Hasura is a separate container. It doesn't share your ASP.NET Core DI container, middleware pipeline, auth policies, or EF Core migrations. Integration seams multiply.
  • OAuth2 scopes require IdP reconfiguration — your existing JWT claims need a namespace transform to work with Hasura. Minor but a real onboarding cost with your identity team.
  • gRPC gap — if any of your L1 consumers are gRPC, Hasura offers nothing natively. A transcoder layer is required.
  • No built-in response cache — for high-frequency financial data reads, you must architect caching outside Hasura.
  • Licence cost at scale — 150 active models on Advanced + Private DDN is ~$5,500/month. For an enterprise this may be justified but must be modelled explicitly.
  • Vendor cloud dependency — full governance tooling (schema registry, CI/CD, console) requires Hasura Cloud or Private DDN. True air-gapped self-host loses significant value.

Hot Chocolate (ChilliCream) — Pros

  • 100% C# / .NET native — your team writes GraphQL resolvers exactly as they write REST controllers. Same DI, same EF Core, same middleware, same debugger. Zero context switch.
  • Existing ASP.NET Core auth works out of the box — AddAuthentication(JwtBearer) + AddAuthorization() + OAuth2 scope policies all apply. No IdP changes, no new auth format.
  • Seamless REST coexistence + gRPC — GraphQL, REST, and gRPC all live in the same ASP.NET Core host. Incremental migration is the default pattern.
  • Full business logic control — MediatR, domain events, saga patterns, outbox, unit of work — all in C#. Complex financial workflows are a first-class citizen.
  • $0 licence + no vendor dependency — MIT licence. Runs on your EKS with no external SaaS dependency. No Private DDN cost.
  • EF Core power — global query filters (built-in multi-tenancy mitigation), migrations, bulk extensions, second-level cache. Full ORM capability.
  • Hot Chocolate Fusion — compose multiple HC subgraphs into a gateway using the GraphQL-Fusion spec (open, MIT). Compatible with Apollo Federation subgraphs. Distributed without vendor lock-in.
  • Trusted documents + cost analysis — enterprise-grade query security built into the framework for both first-party and third-party API scenarios.

Hot Chocolate — Cons

  • CRUD boilerplate per entity — every table still needs a C# model, resolver, DataLoader, and test. Significantly more code than Hasura's introspection model.
  • N+1 requires developer action — DataLoader must be implemented deliberately per relationship. Miss one and you have an N+1 in production.
  • Row-level security is developer-enforced — a missing .Where(x => x.TenantId == ...) in a resolver is a security gap. Requires EF Core global query filters and code review discipline.
  • Schema governance DIY — breaking change detection, schema registry, and immutable API versioning require your own tooling (GraphQL Inspector, Rover CLI, custom CI scripts).
  • AI enablement requires more work — no PromptQL equivalent. Building an AI data agent on HC requires separate MCP plumbing or custom schema-to-prompt context building.

Recommendation for Intelliflo

Given that you are an ASP.NET Core .NET shop with existing REST L1 APIs, SQL Server, EF Core patterns, and financial-grade governance requirements, neither tool is a clear universal winner. The decision depends on what problem you're primarily solving:

Choose Hasura DDN if your priority is…

  • Eliminating data-layer CRUD boilerplate at maximum speed
  • Building AI agents / PromptQL on your financial data (now or soon)
  • Governing a large multi-team data estate with a platform (schema registry, immutable builds, team ownership)
  • Unifying multiple data sources (SQL Server + Snowflake + future NoSQL) in a single API without a .NET service per source
  • Engine-enforced row-level tenancy where accidental bypass is architecturally impossible

Choose Hot Chocolate if your priority is…

  • Maximum leverage of your existing .NET/C# team skills with zero polyglot overhead
  • Incremental migration from your existing REST L1 controllers without splitting infrastructure
  • Complex business logic / DDD / domain events that must live in C# alongside the API
  • gRPC + REST + GraphQL from a single ASP.NET Core process
  • Zero additional licence cost and no vendor cloud dependency
  • Plugging directly into your existing OAuth2/OIDC scope-based auth without IdP changes

Recommended Architecture — The Hybrid Approach

Use Hot Chocolate as your primary L1 GraphQL framework for all ASP.NET Core services, business logic, complex mutations, existing REST/gRPC migration, and scope-based auth. It costs nothing, requires no new skills, and integrates with your entire existing stack immediately.

Introduce Hasura DDN selectively as a dedicated data access layer where you have high-volume, read-heavy, multi-tenant entities that are primarily CRUD — policy data, reference data, reporting data — and where AI agent access or cross-source federation is a near-term requirement. Use it to sit in front of SQL Server and potentially Snowflake without writing resolver code, and compose its subgraph into your HC Fusion gateway.

PoC suggestion: Run both in parallel on the same SQL Server data. Pick one bounded domain (e.g. Portfolios) and implement it in both. Measure schema-to-query time, developer experience, auth integration effort, and operational overhead. The data will make the decision for you.
