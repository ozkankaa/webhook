Architecture Overview — What is Hasura DDN?

Foundation

Hasura DDN (Data Delivery Network) v3 is an immutable, declarative API platform that sits between your clients and data sources. It auto-generates a strongly-typed GraphQL API from your schema and exposes it via a single composable supergraph endpoint. The v3 engine is rewritten in Rust (replacing Haskell in v2), making it stateless and horizontally scalable by design.

Developer Productivity & Experience

Strong Fit
✓ Strong uplift — eliminates ~70–80% of CRUD API boilerplate

What Hasura gives you out of the box

  • Schema introspection — run ddn connector introspect and every table, view, stored procedure and relationship in SQL Server is reflected into typed metadata automatically.
  • Auto-generated operations — queries with filtering, sorting, pagination, aggregations, nested relationships, and mutations (insert/update/delete) without writing a single resolver.
  • GraphiQL console — interactive explorer with schema documentation available locally and on DDN Cloud; no Swagger/OpenAPI to maintain manually.
  • Git-versioned metadata — HML files live in your repo; schema changes are code-reviewed and tracked, not hidden in a dashboard click.
  • Immutable builds — ddn supergraph build create produces a point-in-time snapshot; rollback is trivially re-activating a prior build.
  • Local dev parity — ddn run docker-start spins the exact same engine + connector stack on your laptop via Docker Compose.

Where developer experience has friction

  • Initial learning curve around HML syntax, subgraph structure, and the DDN CLI (~1–2 week ramp for experienced engineers).
  • Complex stored procedures with output cursors or non-standard patterns need manual HML configuration — introspection isn't magic.
  • The console is cloud-hosted by default; offline-only or air-gapped dev requires Private DDN.

AI-Assisted Development & AI Enablement

Strong Fit
✓ Strongly aligned — metadata-first design is ideal for AI code generation

AI-assisted development (developer tooling)

Because Hasura's contract is declarative HML metadata rather than hand-written code, AI coding assistants (GitHub Copilot, Cursor, Claude Code) can:

  • Generate entire subgraph HML files from a natural-language description of a domain entity.
  • Write TypeScript Command implementations for business logic with full type safety inferred from the schema.
  • Produce permission rules from plain-English access requirements (e.g. "users can only see their own orders").
  • Generate GraphQL query documents for integration tests automatically from schema introspection.

AI-enablement for your product (PromptQL / MCP)

Hasura's PromptQL (in active development) is a natural-language query interface that uses the supergraph's semantic model as the AI's grounding truth — no RAG pipeline to build, because your schema already describes your data relationships.

Hasura also ships as a Model Context Protocol (MCP) server, meaning AI agents (Claude, GPT-4o, Copilot agents) can query your supergraph directly as a tool call. This is a first-class citizen in the Hasura DDN roadmap for 2025-26.

Benefits for your architecture

  • Semantic schema — relationships, types, and descriptions in HML give AI agents the context to reason about data without hallucinating column names.
  • Single endpoint — AI agents call one supergraph rather than discovering many REST endpoints.
  • Permission-aware — even AI agents operate under your role-based permission model; no bypass risk.
  • Introspectable — GraphQL schema introspection is a native capability; AI tools can self-discover available queries and mutations.
Strategic alignment: If you are building AI agents that need to read/write financial data, Hasura's supergraph as an MCP server gives you a governed, permission-enforced AI data plane at low incremental cost.

Maintainability, Governance & Migration Path

Strong Fit

Metadata as code

All API surface definitions (models, relationships, permissions, auth config) are .hml files committed to Git. This means:

  • PR-based review for any schema or permission change — full audit trail.
  • CI/CD pipeline validates metadata before it ever reaches production (ddn supergraph build create fails fast on schema errors).
  • Schema registry tracks model-level changes over time (DDN Base+).

Multi-Tenancy

Strong Fit
✓ Row-level multi-tenancy via session variables is a core feature

How it works

Hasura implements multi-tenancy through session variable injection in permissions. Your JWT contains x-hasura-tenant-id (or equivalent), which is then referenced in every model's row-level permission rule:

HML — Row-level tenancy on a Policies model
kind: ModelPermissions
version: v1
definition:
  modelName: Policies
  permissions:
    - role: adviser
      select:
        filter:
          fieldComparison:
            field: tenantId
            operator: _eq
            value:
              sessionVariable: x-hasura-tenant-id

Tenancy patterns supported

Intelliflo context: If your SQL Server uses a tenantId/firmId column pattern (common in wealth management platforms), Hasura's row-level session variable filtering is a drop-in governance layer — no custom middleware needed.

Performance & Operations

Context-Dependent

Query optimisation

  • Query pushdown — the SQL Server connector pushes the entire GraphQL query (including filters, sorts, pagination, joins) down to SQL Server as a single optimised query. No N+1 fetching at the application layer.
  • Native SQL functions — the connector uses ADO.NET and generates SQL leveraging SQL Server's native capabilities (e.g. JSON aggregation, lateral joins where available).
  • Query depth / complexity limits — configurable in DDN to prevent runaway nested queries. Equivalent to field-depth limits and cost analysis in custom GraphQL servers.
  • Allow lists — DDN supports persisted query allow lists (only pre-approved operations are executable in production), which is a key security and performance control.

Caching

Gap to assess: Hasura DDN v3 does not currently ship with a built-in response cache (unlike v2 which had query result caching on Enterprise). Caching must be implemented at the infrastructure layer — API Gateway (e.g. AWS API Gateway + ElastiCache), CDN, or a Redis layer in front of the connector. This is a material consideration for high-frequency read workloads. Validate this during PoC.

Read/Write separation (read replica support)

The SQL Server connector uses a single connection string per connector instance. To route reads to a replica and writes to primary, you have two options:

  • Two connector instances — one pointing at read replica, one at primary. Hasura routes based on operation type (query vs mutation) in metadata.
  • Application-level routing — SQL Server's AlwaysOn AG with read-intent routing can transparently serve reads from replica via a single connection string.

Snowflake / analytics sources

Hasura DDN has an official Snowflake connector (NDC). You can add Snowflake as a second data source in its own subgraph, exposing analytics models alongside transactional models in the same supergraph — a genuinely powerful pattern for unified data access across OLTP and OLAP.

Authorisation — Identity, Claims & Scopes

Strong Fit
✓ JWT + webhook modes natively support your existing IdP claims

Authentication modes

  • JWT mode — Hasura validates a Bearer JWT using a shared secret or JWKs URL (supports RS256, HS256, ES256). Works with Auth0, Azure AD, AWS Cognito, Entra ID, or your own STS.
  • Webhook mode — for non-standard identity flows; Hasura calls your existing auth service to resolve session variables per request.
  • Multiple auth modes — DDN v4 AuthConfig supports primary + alternative modes, enabling gradual migration or multi-client scenarios.

Mapping your claims to Hasura session variables

JWT payload — your IdP must include the Hasura namespace
{
  "sub": "user-uuid",
  "aud": "intelliflo-api",
  "claims.jwt.hasura.io": {
    "x-hasura-default-role":  "adviser",
    "x-hasura-allowed-roles": ["adviser", "admin"],
    "x-hasura-user-id":       "user-uuid",
    "x-hasura-tenant-id":     "firm-xyz",
    "x-hasura-scope":         "portfolio:read policy:write"
  }
}

Any custom x-hasura-* claim is accessible in permission rules as a session variable. Your existing OAuth2 scopes can be embedded as a claim and used to drive field-level or model-level access rules.

Permission model

LevelWhat you controlExample
Model (row-level)Which rows a role can see/mutateadviser sees only their firm's clients
Type (field-level)Which fields within a type are visiblehide NI/NIN from non-admin roles
Command (mutation)Which roles can call a mutation/procedureonly admin can delete a policy
Supergraph (subgraph)Subgraph visibility per collaborator roleseparate teams own separate domains

Integration path with your IdP

If your identity provider is Azure Entra ID or a custom STS, the recommended approach is to add an Auth0 Action / Azure AD claims transform that injects the claims.jwt.hasura.io namespace into your JWT. This is a one-time IdP configuration, not a code change in Hasura.

Scope-based permissions: Hasura doesn't have native OAuth2 scope enforcement (e.g. "portfolio:read") — it works through roles. The recommended pattern is to map scopes → roles in your IdP's token transform, or use the webhook mode where your auth service resolves the full session variable set from scopes.

Bulk & Batch Operations

Partial — Validate in PoC
⚡ Supported, but with constraints worth validating

What's available

  • Bulk inserts — the SQL Server connector supports insert_many mutations accepting an array of objects in a single GraphQL operation.
  • Batch GraphQL requests — Hasura supports HTTP batching (array of GraphQL operations in a single POST), executed in parallel.
  • Native queries / stored procedures — for complex bulk operations (e.g. bulk recalculation), expose your existing SQL Server stored procedures as Commands in Hasura. This is the recommended pattern for keeping heavy set-based SQL logic inside the database.
  • TypeScript/Python Commands — for orchestration logic that fans out to multiple models, implement as a server-side function that Hasura exposes as a single GraphQL mutation.

Constraints to validate

  • Bulk updates with heterogeneous payloads (update each row differently) are less ergonomic in GraphQL vs a dedicated REST bulk endpoint — may still need a Command/stored-proc pattern.
  • Transaction semantics across multiple mutations in a single batch: Hasura executes sequential mutations in a transaction for PostgreSQL; verify this behaviour for the SQL Server connector.
  • Large batch sizes (e.g. 10,000+ rows) — validate memory and timeout behaviour under load.
PoC action: Include a bulk insert (1,000 rows) and a stored procedure call as Command in your PoC. This validates both the connector's batch capability and your existing SP migration path in one test.

Cost & Pricing

Predictable

Pricing model — "active models"

Hasura DDN prices per active model per month. An active model is any entity (table/view) that receives ≥ 1,000 requests/month. This makes costs predictable and decoupled from traffic volume — you pay for the complexity of your API surface, not for load.

Compare to: engineering cost of maintaining bespoke REST L1 services (developer time, infrastructure, testing overhead). For 150 models, Hasura Advanced likely saves significantly more than $4,500/mo in engineering overhead. Model the break-even with your team's fully-loaded rate.

Deployment options

  • Hasura Cloud (DDN) — fully managed, Hasura-hosted data plane on their CDN. Data connector runs in your VPC, engine on Hasura Cloud. Requires DB to be reachable from internet OR use tunnelling.
  • Private DDN — data plane deployed in your own AWS VPC or a dedicated Hasura-managed VPC. Full network isolation. Recommended for financial data.
  • Self-hosted (open source) — run the v3 engine and connectors entirely on your EKS cluster. No Hasura Cloud dependency. No built-in console beyond local dev; you manage upgrades.

Local PoC — SQL Server Setup Guide

Step-by-Step
PoC objective: Validate developer workflow, SQL Server connector capabilities, JWT auth with tenant isolation, stored procedure as Command, and bulk insert — all running locally on Docker.

Prerequisites

  • Docker Desktop (or Rancher Desktop — note: Rancher may intercept kubeconfig paths as symlinks; see WSL2 config)
  • Node.js 20+ (for TypeScript Commands)
  • DDN CLI — download from https://hasura.io/docs/3.0/cli/installation/
  • A Hasura DDN account (free tier is sufficient for PoC)

Step-by-step setup

Authenticate DDN CLI

ddn auth login   # opens browser OAuth2 flow

Scaffold a new project

ddn supergraph init intelliflo-l1 --with-starter-project
cd intelliflo-l1

This creates the directory structure: app/ (subgraph), engine/, globals/.

Start a local SQL Server container

docker run -e "ACCEPT_EULA=Y" \
  -e "MSSQL_SA_PASSWORD=StrongP@ss1!" \
  -p 1433:1433 \
  --name sqlserver-poc \
  -d mcr.microsoft.com/mssql/server:2022-latest

# Seed a sample financial schema
docker exec -i sqlserver-poc /opt/mssql-tools18/bin/sqlcmd \
  -S localhost -U sa -P "StrongP@ss1!" -No < seed.sql

Seed SQL — create sample L1 schema

seed.sql
-- Multi-tenant financial schema
CREATE DATABASE intelliflo_poc;
GO
USE intelliflo_poc;
GO

CREATE TABLE Tenants (
  TenantId    UNIQUEIDENTIFIER PRIMARY KEY DEFAULT NEWID(),
  FirmName    NVARCHAR(200) NOT NULL
);

CREATE TABLE Clients (
  ClientId    UNIQUEIDENTIFIER PRIMARY KEY DEFAULT NEWID(),
  TenantId    UNIQUEIDENTIFIER NOT NULL REFERENCES Tenants(TenantId),
  FullName    NVARCHAR(200) NOT NULL,
  NiNumber    NVARCHAR(20),
  CreatedAt   DATETIME2 DEFAULT GETUTCDATE()
);

CREATE TABLE Portfolios (
  PortfolioId UNIQUEIDENTIFIER PRIMARY KEY DEFAULT NEWID(),
  ClientId    UNIQUEIDENTIFIER NOT NULL REFERENCES Clients(ClientId),
  TenantId    UNIQUEIDENTIFIER NOT NULL,
  Name        NVARCHAR(100) NOT NULL,
  Valuation   DECIMAL(18,2),
  ValuedAt    DATETIME2
);

-- Stored procedure for bulk portfolio valuation update
CREATE PROCEDURE sp_BulkUpdateValuations
  @ValuationData NVARCHAR(MAX)  -- JSON array
AS BEGIN
  UPDATE p SET p.Valuation = v.Valuation, p.ValuedAt = GETUTCDATE()
  FROM Portfolios p
  JOIN OPENJSON(@ValuationData) WITH (
    PortfolioId UNIQUEIDENTIFIER '$.portfolioId',
    Valuation   DECIMAL(18,2)   '$.valuation'
  ) v ON p.PortfolioId = v.PortfolioId;
END;
GO

-- Sample data
INSERT INTO Tenants VALUES (NEWID(), 'Smith Financial');
INSERT INTO Tenants VALUES (NEWID(), 'Jones Wealth');
GO

Add the SQL Server connector

# With context set to your 'app' subgraph
ddn connector init my_sqlserver \
  --subgraph app \
  --hub-connector hasura/sqlserver

When prompted, enter the ADO.NET connection string for CONNECTION_URI:

Server=localhost,1433;Database=intelliflo_poc;User Id=sa;Password=StrongP@ss1!;TrustServerCertificate=true;

Introspect the database

ddn connector introspect my_sqlserver

This generates app/connector/my_sqlserver/configuration.json — all tables, views, and stored procedures reflected.

Track models (tables as GraphQL types)

ddn model add my_sqlserver Clients
ddn model add my_sqlserver Portfolios
ddn model add my_sqlserver Tenants

HML files are created at app/metadata/Clients.hml etc., with auto-detected types and relationships from foreign keys.

Add a Command for the stored procedure

ddn command add my_sqlserver sp_BulkUpdateValuations

This exposes the stored proc as a GraphQL mutation. Callers pass the JSON valuation payload directly.

Add row-level permissions (multi-tenancy)

app/metadata/Clients.hml — add permissions block
kind: ModelPermissions
version: v1
definition:
  modelName: Clients
  permissions:
    - role: admin
      select:
        filter: null           # admin sees all
    - role: adviser
      select:
        filter:
          fieldComparison:
            field: tenantId
            operator: _eq
            value:
              sessionVariable: x-hasura-tenant-id
    - role: adviser
      select:
        fields:
          # Hide NI number from adviser role
          - clientId
          - fullName
          - createdAt

Recommended approach

Complement, then replace. Introduce Hasura DDN as a parallel layer for net-new domain entities and greenfield subgraphs. Migrate existing L1 REST APIs entity-by-entity as the PoC builds confidence. Retain complex business logic services; expose them through Hasura as Commands or federate them via the supergraph. Do not attempt a big-bang replacement.
